소프트웨어 테스트 필요성
1. 프로그램에 잠재된 오류를 발견 이를 수정하여 올바른 프로그램을 개발
2. 프로그램 실행 전에 동료 검토, 인스펙션 등을 통해 오류를 사전에 발견하는 예방 차원
3. 사용자의 요구사항 및 기대 수준을 만족하도록 반복적인 테스트를 거쳐 제품의 신뢰도를 향상하는 품질 보증을 위해 필요

소프트웨어 테스트 원리
1. 결함이 존재함을 밝히는 활동
2. 완벽하게 테스팅하려는 시도는 불필요한 시간과 자원낭비
3. 조기 테스트 설계시 장점이 있음
4. 적은 수의 모듈에서 대다수의 결함이 발견함
5. 동일한 테스트 케이스에 의한 반복적 테스트는 새로운 버그 찾지못함
6. 소프트웨어의 성격에 맞게 테스트 실시
7. 요구사항을 충족시켜주지 못한다면 결함이 없다고 해도 품질이 높다고 볼 수 없다.

소프트웨어 테스트 산출물
1. 테스트 목적봐 범위 정의 등 테스트 수행을 계획한 문서

3. 테스트를 위한 설계 산출물로 테스트 항목 명세서
4. 테스트 케이스를 실행환경에 따라 구분해 놓은 테스트 케이스의 집합
5. 애플리케이션의 테스트 되어야 할 기능 및 특징, 테스트가 필요한 상황을 작성한 문서
6. 테스트 케이스의 실행 순서를 작성한 문서
7. 테스트 결과를 정리한 문서

2. 도구의 지원을 받아서 하는 방법

동적 테스트 분류

2. 프로그램 외부 사용자의 요구사항 명세를 보면서 수행하는 테스트

3. 결정 포인트 내의 각 개별 조건식이 적어도 한 번은 참과 거짓의 결과가 되도록 수행하는 테스트 커버리지
4. 전체 조건식뿐만 아니라 개별 조건식도 참 한번,거짓 한번 결과가 되도록 수행하는 테스트 커버리지
5. 개별 조건식이 다른 개별 조건식에 영향을 받지 않고 전체 조건식에 독립적으로 영향을 주도록 함
6. 결정 조건 내 모든 개별 조건식의 모든 가능한 조합을 100% 보장하는 커버리지
7. 수행 가능한 모든 경로를 테스트하는 기법
8. 프로그램 제어 구조를 그래프 형태로 나타내어 내부로직을 테스트하는 기법
9. 제어 흐름 그래프에 데이터 사용현황을 추가한 그래프를 통해 테스트하는 기법
