개별적인 모듈을 테스트하는 것

목 객체 유형
1. 테스트 할 때 객체만 필요하고 해당 객체의 기능까지는 필요하지 않은 경우에 사용
2. 제어 모듈이 호출하는 타 모듈의 기능을 단순히 수행하는 도구
3. 테스트 대상 하위 모듈을 호출하고 파라미터를 전달하고 모듈 테스트 수행 후의 결과를 도출
4. 주로 테스트 대상 클래스와 협력하는 클래스로 가는 출력을 검증하는 데 사용
5. 실제 협력 클래스의 기능을 대체해야 할 경우 사용하는 객체

각 모듈 간의 인터페이스 관련 오류 및 결함을 찾아내기 위한 체계적인 테스트 기법

통합 테스트 분류
1. 모든 모듈을 동시에 통합 후 테스트 수행
2. 최하위 모듈로부터 점진적으로 상위 모듈과 함께 테스트, 테스트 드라이버 필요함
3. 최상위 모듈부터 하위 모듈들을 통합하면서 테스트, 테스트 스텁이 필요함
4. 상위는 하향식, 하위는 상향식 테스트를 거치는 테스트

테스트 도구를 활용하여 반복적인 테스트 작업을 스크립트 형태로 구현함으로써 테스트 시간 단축과 인력 투입 비용 최소화, 효율적인 테스트 수행하도록 하는 도구

테스트 자동화 도구 유형
1. 만들어진 애플리케이션을 실행하지 않고 분석하는 도구
2. 테스트를 위해 작성된 스크립트를 실행하고 작성된 스크립트는 각 스크립트마다 특정 데이터와 테스트 수행 방법을 포함
3. 애플리케이션의 처리량, 응답 시간, 경과 시간, 자원 사용률에 대해 가상의 사용자를 생성하고 테스트를 수행
4. 테스트 계획 및 관리를 위한 테스트 관리 도구, 테스트 수행에 필요한 데이터와 도구를 관리하는 형상 관리 도구, 테스트 결함에 대해 관리하거나 협업을 지원하기 위한 결함 추적/관리 도구 등

테스트 실행 도구 종류
1. 테스트 데이터를 스프레드 시트에 저장
2. 일반적으로 테스트를 수행할 동작을 나타내는 키워드와 테스트 데이터를 스프레드시트에 저장

애플리케이션 컴포넌트 및 모듈을 테스트하는 환경의 일부분으로 테스트를 지원하기 위한 코드와 데이터를 뜻함

테스트 하네스 구성요소
1. 테스트 대상 하위 모듈을 호출하고 파라미터를 전달, 모듈 테스트 수행 후의 결과를 도출
2. 제어 모듈이 호출하는 타 모듈의 기능을 단순히 수행하는 도구
3. 테스트 대상 컴포넌트나 모듈, 시스템에 사용되는 테스트 케이스의 집합
4. 입력값, 실행조건, 기대 결과 등의 집합
5. 자동화된 테스트 실행 절차에 대한 명세
6. 사용자의 행위를 조건부로 사전에 입력해 두면 상황에 예정된 행위를 수행하는 객체

테스트를 언제 어떤 상황에서 종료할 것인지를 결정하는 기준

테스트 리포팅 종류
1. 모든 테스트가 완료되면 테스트 계획과 테스트 케이스 설계부터 단계별 테스트 시나리오 테스트 결과까지 모두 포함된 문서 작성
2. 테스트 계획, 소요 비용, 테스트 결과로 판단 가능한 대상 소프트웨어의 품질 상태를 포함한 문서 작성
3. 정량회된 품질 지표인 테스트 성공륭, 테스트 커버리지, 발생한 결함의 수와 결함의 중요도 등이 포함
4. 결함과 관련한 사항을 중점적으로 기록하며 결함 내용 및 자원 정보, 결함의 재현 순서를 상세하게 기록
5. 단계별 테스트 종료 시 테스트 실행 절차를 리뷰하고 결과에 대한 평가를 수행하고 그 결과에 따라 실행 절차를 최적화하여 다음 테스트에 적용

결함의 분류
1. 비정상적인 종료/중단, 응답 시간 지연 등 애플리케이션 환경과 데이터베이스 처리에서 발생하는 결함
2. 사용자의 요구사항 미반영/불일치, 스크립트 에러, 등의 단계에서 발생한 결함
3. 응용 프로그램의 ui 비일관성, 부정확한 메시지 등 사용자 화면 설계에서 발생한 결함
4. 기획자, 사용자, 개발자 간의 의사소통과 기록이 원활하지 않은 경우에 발생하는 결함

결함 분석 방법
1. 결함의 원인을 찾기 위해 결함을 발생시킨 입력값, 절차,환경을 명확히 파악하는 방법
2. 입력값 절차, 환경 중 어떤 요소가 결함 발생에 영향을 미치는지 분석하는 방법
3. 결함 발생에 영향을 주는 요소를 최대한 일반화 시키는 방법

결함은 ~,~,~,~ 시 유입되는 결함이 있다.

얼마나 치명적인지를 나타내는 척도이다

결함 심각도별 분류
1. 기능이나 제품의 테스트를 완전히 방해하거나 못하게 하는 결함
2. 기능이 기대와 많이 다르게 동작하거나 기능이 해야할 것을 못하는 결함
3. 제품이나 프로그램이 특정 기준을 충족하지 못하거나 전체에 영향을 주지 않는 일부 기능이 부자연스러운 결함
4. 사용상의 불편함을 유발하는 결함
5. 사소한 버그라고 하며 기능에는 영향이 없지만 수정되어야 하는 결함

결함 생명주기
1. 발견 결함을 분석 -> 보고
2. 처리방안 검토
3. 결함 수정할 개발자 결정
4. 해결 처리한 상태
5. 결함 처리 합당한지 검증
6. 수정이 이루어졌다고 판단되어 종료
7. 결함 정확히 수정 x시 다시 수정요구
8. 결함을 고답로 수정 안하고 다음 릴리스에서 해결하기로 연기된 상태

1. 일반적으로 사람에 의해 생성된 실수
2. 소프트웨어 제품에 포함된 문제
